1. HTML tags

Every tag that appears in index.html or a project page. Beyond the standard semantics, each entry notes how this project actually uses the tag.

Document scaffolding

<!doctype html>
HTML5 document declaration. Must be the very first line — without it browsers fall into quirks mode and many layout rules behave differently.
<html lang>
Root element. lang="en" tells screen readers and search engines the document language.
<head>
Container for metadata that isn't rendered: title, meta tags, fonts, stylesheets.
<meta>
Single-attribute metadata. Used here for character encoding (charset), responsive viewport (name="viewport"), and SEO description.
<title>
Browser tab label and default bookmark name.
<link>
External resource. Two uses: rel="stylesheet" for CSS, and rel="preload" as="font" to prefetch web fonts before the first paint so no FOUT (Flash Of Unstyled Text).
<body>
Holds everything that renders.

Semantic structure

<header>
Top region. Holds the fixed nav and its progressive-blur backdrop.
<nav>
Primary navigation. Contains the site links (About / Projects / Music).
<main>
The single main landmark for the page. There must be exactly one. Screen readers can jump straight to it.
<section>
A thematic grouping with its own heading. Hero, About, Projects, Music, etc.
<article>
Self-contained content; used here for each .paper sheet on a project page.
<footer>
Closing region. (Site footer.)
<figure> / <figcaption>
An image with its caption, kept together semantically. Used for every polaroid on project pages.

Content & text

<h1>
Top-level heading. One per page (visually hidden on the homepage because the SVG name takes its place).
<h2>
Section heading: About, Projects, Music, Currently working on…
<p>
Paragraph. The atomic block of body copy.
<ul> / <ol>
Unordered / ordered list. The nav links are an <ul>; the projects timeline is an <ol> (the order matters, it's chronological).
<li>
One list item. Each project is one <li>.
<a href>
Hyperlink. The internal jumps (#about) plus the cassettes that link to project pages.
<br>
Explicit line break inside a paragraph (used sparingly in the About text).
<b>
Stylistically bold inline text without semantic emphasis (used inside .meta on music sleeves).

Media & presentation

<img src alt>
Raster/vector image. alt is mandatory — empty alt="" for purely decorative images (camera, portrait), descriptive alt for content images (project previews).
<div>
Generic block grouping with no semantics. Used for layout shells (.about-stage, .nav-blur, etc.).
<span>
Generic inline grouping. Used for the progressive blur layers — each <span> carries inline CSS custom properties (--b, --t, --e, --m) that the stylesheet reads.

SVG (used in the animated name)

<svg viewBox>
Inline vector graphic. viewBox defines the internal coordinate system; the SVG then scales fluidly via the .draw-name { width: min(60vw, 32.5rem) } rule.
<g>
Group element. Used to apply a single SVG filter (the glow halo) to all strokes at once.
<path d>
The geometry. Each letter stroke is one <path> with a vector d attribute and a pathLength="1" normalisation (see §4).
<defs>
Reusable definitions. The gradients, filter and clip paths sit inside <defs>; nothing inside is rendered directly, only referenced by id.
<filter>
SVG filter primitive container. Used for the soft glow behind each stroke.
<feGaussianBlur>, <feMerge>, etc.
Filter primitives composing the glow.
<linearGradient> / <stop>
Defines the colour gradients on individual strokes.

3. ★ Section-by-section walkthrough

The site is one long <main> made of five sections plus a fixed header and a footer. Each section uses a different mix of layout primitives. This section explains, in order, what the HTML looks like and what each piece does.

3.1 <header class="site-header"> — the fixed nav

<header class="site-header">
    <div class="nav-blur" aria-hidden="true">
        <span style="--b: 2px;  --t: 45%; --e: 100%"></span>
        <span style="--b: 4px;  --t: 38%; --e: 88%"></span>
        <!-- … 4 more progressive layers -->
    </div>
    <nav class="nav">
        <a class="brand" href="#top">antoineview</a>
        <ul class="nav-links">
            <li><a href="#about">About</a></li>
            <li><a href="#projects">Projects</a></li>
            <li><a href="#music">Music</a></li>
        </ul>
    </nav>
</header>

Two children: a decorative .nav-blur backdrop and the real <nav>. The aria-hidden="true" tells screen readers to skip the blur (it's purely visual). The header itself is position: fixed; top: 0; left: 0; right: 0 so it stays anchored to the viewport on every scroll. Each blur <span> carries its parameters as inline CSS variables (--b=radius, --t=mask solid-end, --e=mask transparent-end), so one CSS rule renders six different progressive blur layers.

3.2 <section class="hero" id="top"> — the scroll-driven name

<section class="hero" id="top">
    <div class="hero-pin">
        <div class="hero-imgs" aria-hidden="true">
            <img class="hero-pic" src="…/paper-city.jpeg" />
            <!-- 7 more drifting pictures -->
        </div>
        <div class="edge-blur" aria-hidden="true">…</div>
        <div class="edge-fade" aria-hidden="true"></div>
        <h1 class="visually-hidden">antoine</h1>
        <svg class="draw-name" viewBox="0 0 854 163">
            <path style="--i: 5" pathLength="1" d="…" />
            <!-- 23 more path strokes -->
        </svg>
        <p class="scroll-cue">scroll</p>
    </div>
</section>

Two-layer structure. The outer .hero is the scroll runway — it's 300vh tall when scroll-driven animations are supported, so you can scroll three viewports' worth while looking at it. The inner .hero-pin is position: sticky; top: 0; height: 100vh — it pins to the top of the viewport for the duration of that runway. While the runway is in view, scrolling advances the SVG path animation (see §5) without moving the visible content.

What lives inside the pin.

  • .hero-imgs: the drifting photographs. A CSS grid of images, each running a @keyframes drift animation on a 36-second loop with staggered animation-delay.
  • .edge-blur & .edge-fade: progressive blur and a white gradient at the viewport edges, so the drifting photos dissolve into the page background rather than cutting off at a hard rectangle.
  • .visually-hidden <h1>: a screen-reader-only "antoine" so the page has a real H1 even though the visible logotype is an SVG.
  • .draw-name: the SVG itself — 24 paths drawing each letter stroke. See §5.

3.3 <section id="about" class="about"> — sticky portrait + scrolling text

<section id="about" class="about">
    <h2 id="about-heading">About</h2>
    <div class="about-stage">
        <img class="about-portrait" src="…/portrait.jpeg" />
        <div class="about-veil">
            <span class="about-veil-white"></span>
            <span class="about-veil-blur" style="--b: 2px;  --m: 100%"></span>
            <!-- 5 more progressive blur layers -->
        </div>
        <img class="about-camera" src="…/camera.png" />
        <div class="about-text">
            <p>Hi! I'm Antoine</p>
            <!-- more paragraphs -->
        </div>
    </div>
</section>

The trick: four layers in the same grid cell.

.about-stage {
    display: grid;
    grid-template-columns: var(--about-w);
    grid-template-areas: "stage";
    justify-content: center;
    min-height: 220vh;
}
.about-portrait, .about-veil, .about-camera, .about-text {
    grid-area: stage;      /* all four land in the same cell */
}

By declaring a single named area and putting all four elements into it, they overlap. z-index then controls who paints on top: portrait (1) → white veil (2) → camera (3) → text (4).

The scroll choreography. The .about-stage is 220vh tall (a long runway). Inside it:

  • The portrait is position: sticky; top: 25%. It pins a quarter of the way down the viewport while the stage scrolls past.
  • The veil and camera are in normal flow (no sticky). They scroll up off-screen as you advance.
  • The text is in normal flow too, but with padding-block-start: calc(var(--about-w) + 5vh) so its first paragraph begins below the bottom of the camera. As the text scrolls up past the still-sticky portrait, it reads as a caption track over the trees.

The white veil hides the portrait until the camera (and the veil with it) has scrolled away. Then the portrait alone is left, with the text scrolling over it.

Full-bleed. The .about uses width: 100vw; margin-inline: calc(50% - 50vw) to break out of main { max-width: 45rem } and span the entire viewport — necessary because the camera and portrait want all the horizontal room they can get.

4. ★ Scroll-driven animations

Used in: §3.2 (hero name draws as you scroll). Declares a timeline channel on §3.3 (about) for downstream effects.

The hero name draws itself as you scroll, and the about section advertises a timeline downstream effects can hook into. Both use the brand-new Scroll-driven Animations spec, which connects a regular @keyframes animation to scroll progress instead of wall-clock time.

The three pieces

  • view-timeline: --name block — declares that this element exposes a timeline named --name that advances from 0 when the element first enters the viewport to 1 when it has just left, measured along the block axis (vertical for default writing modes).
  • animation-timeline: --name — binds an animation to that timeline. Progress through the keyframes now follows scroll progress, not seconds.
  • animation-range: contain X% contain Y% — restricts which slice of the timeline the animation occupies. contain means "the element is fully contained in the viewport," so contain 0% contain 100% runs the animation only while the element is fully on screen.

The hero name, in detail

/* Outer: 300vh-tall scroll runway exposes a timeline */
@supports (animation-timeline: view()) {
    .hero {
        height: 300vh;
        view-timeline: --hero block;
    }

    .hero-pin {
        position: sticky; top: 0; height: 100vh;
    }

    /* Each letter stroke binds to the same timeline */
    .draw-name path {
        animation-name: draw-stroke;
        animation-timeline: --hero;
        animation-duration: auto;           /* duration is the timeline */
        animation-delay: 0s;
        animation-timing-function: linear;
        animation-fill-mode: both;
        /* And each stroke gets its own 1/24 slice */
        animation-range:
            contain calc((var(--i) - 1) / 24 * 100%)
            contain calc((var(--i))     / 24 * 100%);
    }
}

The strokes are numbered 1–24 via an inline CSS variable --i on each <path>. Each stroke draws across its own equal 1/24 slice of the contain phase, so by the time the section has fully scrolled past, every letter has appeared in order.

Fallback for browsers without scroll-timeline

Outside the @supports block, the same keyframes run on a normal time-based animation with a per-stroke animation-delay: calc((var(--i) - 1) * 170ms). Same visual result, just untethered from scroll. This is graceful degradation: the modern path is opt-in via feature query.

The about timeline

.about {
    view-timeline: --about block;
}

The about section declares a timeline named --about so future scroll-driven effects (e.g. crossfading the portrait, advancing parallax) can attach by name without re-measuring scroll position from JavaScript.

Mental model

Think of view-timeline as the element publishing its own progress as a 0→1 channel that anyone in the cascade can subscribe to. animation-timeline is the subscription. animation-range is a window filter on that subscription. The keyframes themselves don't change — they just play to a different clock.

This replaces a category of JavaScript that used to listen on scroll, compute positions, and write transforms every frame. It's declarative, off the main thread, and respects prefers-reduced-motion.

5. ★ SVG path animation

Each letter in the SVG name draws itself stroke-by-stroke. The technique combines three SVG/CSS features.

1. pathLength

Without pathLength, the dash pattern is in user units — so a long stroke and a short stroke would need different dash values to look the same. Setting pathLength="1" on the <path> tells the browser "pretend this path has length 1," normalising every stroke to the same length regardless of its actual geometry.

2. stroke-dasharray and stroke-dashoffset

A stroke can be drawn as a dashed line by setting stroke-dasharray. With pathLength="1" + stroke-dasharray: 1, you've drawn one dash exactly the length of the whole path: a single, solid line.

stroke-dashoffset shifts where that dash starts. At offset 1 (= one whole path length away), the dash is entirely off the end → invisible. At offset 0, it's perfectly aligned → fully visible.

.draw-name path {
    stroke-dasharray: 1;
    stroke-dashoffset: 1;    /* hidden */
    visibility: hidden;
    animation: draw-stroke 170ms linear forwards;
}

@keyframes draw-stroke {
    0%   { stroke-dashoffset: 1; visibility: hidden;  }
    100% { stroke-dashoffset: 0; visibility: visible; }
}

3. Ordering with a custom property

Each <path> is tagged inline with style="--i: 8" (1–24, the writing order). The CSS then derives both:

  • The fallback delay: animation-delay: calc((var(--i) - 1) * 170ms) — letter --i waits its turn.
  • The scroll-timeline range: animation-range: contain calc((--i - 1)/24 * 100%) contain calc(--i/24 * 100%) — letter --i occupies its slice of the scroll runway.

Why visibility: hidden at the start?

Subpixel anti-aliasing can leak a faint sliver of the stroke even at dashoffset: 1. Toggling visibility through the keyframes guarantees a clean start: nothing renders until the first frame where the dashoffset is meaningfully less than 1.

6. ★ Sticky & fixed positioning

"Sticky" is the most useful and most misunderstood positioning value in CSS. It's the engine behind every scroll choreography in this site, so it gets a deep dive.

The mental model

A sticky element behaves like position: relative most of the time. It flows with the document, takes up its natural space, and scrolls along with its parent. But when the scroll position would cause the element to leave the viewport on its nearest scrolling ancestor, the browser switches it to behave like position: fixed at the offset you specified (e.g. top: 0). The element appears to "stick" to that offset.

The pin lasts only as long as the element's containing block (its nearest ancestor with a layout role) is still visible. Once the bottom of the containing block scrolls past the sticky offset, the element un-pins and scrolls away with its parent.

The three required ingredients

  1. An offset. You must declare at least one of top, bottom, left, right. Without one, sticky silently degrades to position: relative and nothing happens.
  2. A parent taller than the sticky element. The parent needs scroll headroom for the element to stick within. If parent and child are the same height, they leave the viewport together — no sticking ever happens.
  3. No scroll-blocking ancestor. Any ancestor with overflow: hidden, overflow: auto, or overflow: scroll becomes the new scrolling root for the sticky. If that ancestor doesn't itself scroll, the sticky inside it can't either.

The three sticky uses in this site

1. The hero pin — pinning the whole hero for 300vh of scroll

.hero      { height: 300vh; view-timeline: --hero block; }
.hero-pin  { position: sticky; top: 0; height: 100vh; }

The outer .hero is 300vh tall — it gives you three viewports' worth of scroll room. The inner .hero-pin is 100vh tall (one viewport) and position: sticky; top: 0. The result: as you scroll the 300vh runway, the inner pin stays glued to the top of the viewport. Visually nothing moves, but you're advancing through scroll progress. The scroll-driven SVG animation uses that progress to draw the name (see §4–5).

2. The projects TV — pinning a side panel while the timeline scrolls past

.projects-grid { display: grid; grid-template-columns: 1fr 1fr; align-items: start; }
.frame         { position: sticky; top: var(--head-top); }

A two-column grid: timeline on the left, TV on the right. The TV's container (.frame) sticks to top: var(--head-top) (5rem, leaving room for the fixed nav). As long as the parent .projects-grid is on screen, the TV stays parked at that offset while the timeline column scrolls past it. Hovering a cassette in the timeline triggers a :has() rule that swaps the visible .tv-pic inside the still-sticky TV.

Why align-items: start? The default grid alignment is stretch, which would make the .frame grow to the timeline's height. A stretched sticky has no scroll headroom (it fills its row exactly) and can't pin. align-items: start lets each column be its natural height.

3. The about portrait — pinning behind scrolling text and layered images

.about-stage    { min-height: 220vh; }
.about-portrait { position: sticky; top: 25%; }

The portrait sits in the same grid cell as the white veil, the camera and the text. It's position: sticky; top: 25% — pinning a quarter of the viewport down from the top. The runway is 220vh tall, so as you scroll:

  1. For the first stretch, the portrait flows with the runway alongside the camera and veil.
  2. Once the portrait's natural position reaches 25% from the top, it pins.
  3. The veil and camera (not sticky) keep scrolling up and disappear off the top of the viewport.
  4. The text (not sticky) scrolls up across the now-uncovered, still-pinned portrait — reading like captions over the trees.
  5. Eventually the bottom of .about-stage hits 25% from the top, the portrait un-pins, and the whole section scrolls away.

Sticky uses in this project

1. The hero pin (.hero-pin)

position: sticky; top: 0; height: 100vh inside a .hero { height: 300vh } runway. As you scroll the 300vh section, the inner 100vh content "sticks" to the top of the viewport for the duration. This is what gives the scroll-driven SVG animation room to play: you can scroll 300vh while looking at the same fixed-feeling hero, and the scroll progress drives the path reveals.

2. The TV in the projects grid (.frame)

position: sticky; top: var(--head-top) — the TV stays parked under the nav while the timeline of cassette items scrolls past it. On hover of any timeline item, the corresponding .tv-pic swaps in via opacity transition.

3. The about portrait (.about-portrait)

position: sticky; top: 25% inside a 220vh-tall .about-stage. The portrait pins a quarter of the way down the viewport as the white veil/camera/text layers scroll over it.

position: fixed

Only one element on the site uses it: .site-header (left: 0; right: 0 so it spans exactly the viewport regardless of scroll). Unlike sticky, fixed ignores its containing block and pins to the viewport itself.

Common sticky gotchas

  • An ancestor with overflow: hidden kills sticky in browsers that treat it as a new scroll container. overflow: clip doesn't — that's why we use clip on <html> and <body>: it suppresses horizontal scroll without breaking sticky descendants.
  • A flex/grid ancestor with align-items: stretch can make the sticky's container shrink to fit the sticky, so it never has room to "stick." Use align-items: start on the parent grid (which is what .about-stage does for the portrait/camera/veil layers).
  • Sticky doesn't work without a real top (or bottom/left/right) — without one, it falls back to position: relative.

7. At-rules

@font-face

@font-face {
    font-family: "BBB Karrik";
    src: url("../fonts/BBB_Karrik.woff2") format("woff2"),
         url("../fonts/BBB_Karrik.woff")  format("woff");
    font-weight: 400;
    font-style: normal;
    font-display: swap;
}

Registers a custom font. font-display: swap tells the browser "show fallback text immediately, swap the real font in once it loads." Combined with <link rel="preload" as="font" crossorigin> in the document head, the font is requested at the same time as the HTML and arrives before first paint.

@keyframes

@keyframes drift {
    0%, 100% { translate: 26vw 0vh; }
    25%      { translate: 0     18vh; }
    50%      { translate: -26vw 0vh; }
    75%      { translate: 0    -18vh; }
}

Defines named animation timelines. Then any element binds with animation: drift 36s linear infinite. We use keyframes for the hero drift, the vinyl spin, the SVG draw-stroke and a few others.

@media

Conditional rule blocks. Three breakpoints in this site:

  • @media (max-width: 40em) — phones (≤640px). Single-column projects grid, mobile-only TV veil, condensed timeline, smaller about composition.
  • @media (min-width: 40.0625em) and (max-width: 64em) — small tablets (~641–1024px). In-between about sizing, narrower hero images grid.
  • @media (prefers-reduced-motion: reduce) — disables the hero drift, the SVG draw-stroke (just shows the final state), the vinyl spin, etc. Accessibility default.

@supports

@supports (animation-timeline: view()) {
    /* modern, scroll-driven path */
}

Feature query. Only applies the enclosed rules if the browser actually understands the property/value combination. Lets us layer the scroll-driven version on top of a time-based fallback without a JavaScript detect.

8. Recurring techniques

Custom properties as a parameter API

Inline style="--b: 16px; --t: 48%; --e: 68%" on each blur span lets one CSS rule render six layered backdrop-filters with different radii and fade points, without any JS. Same idea on the SVG paths (--i: 8 indexes a stroke into the animation sequence) and the about token (--about-w drives camera width, portrait width, veil height and text padding all at once).

Full-bleed inside a constrained <main>

.about {
    width: 100vw;
    margin-left:  calc(50% - 50vw);
    margin-right: calc(50% - 50vw);
}

Breaks an element out of main { max-width: 45rem } to the full viewport width, recentered. Combined with html, body { overflow-x: clip } to prevent the resulting overflow from triggering horizontal scroll.

Layered grid cell + z-index

The about composition stacks portrait, white veil, camera and text in the same grid cell via grid-area: stage on all four, then uses z-index to control paint order. No absolute positioning needed.

Progressive backdrop-blur

Six <span> layers each with backdrop-filter: blur(var(--b)) at radii 2, 4, 8, 16, 28, 44 px, each masked by its own linear-gradient. The sum is a smooth blur ramp rather than a single hard-edged blur boundary. Used in the navbar, hero edge fade, and about veil.

:has() for parent-aware hover

.k7-link:has(.player-front:hover) .k7-axis { animation-play-state: running } — when the cover is hovered, spin the cassette axes; when only the cassette body is hovered, don't. The :has() selector lets a parent rule react to a descendant's state without JavaScript.

Mobile-only DOM repurposing via order

Same DOM order across all devices. On mobile, .frame { order: -1 } pulls the TV above the timeline; combined with display: grid; grid-template-columns: 1fr the layout reads top-to-bottom in a sensible way without restructuring the markup.
